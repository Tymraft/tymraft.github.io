Tymraft Document Viewer Platform

1 Overview

The Tymraft team currently lacks a centralised location for internal documentation. Documents are scattered across email threads, local drives, and ad-hoc shared folders. This creates friction when onboarding new team members, referencing past decisions, or maintaining living documents over time.

This BRD defines the requirements for a lightweight, Git-backed document viewer hosted at tymraft.github.io. The platform will allow team members to publish Markdown and HTML documents to a GitHub repository and immediately access them through a clean, navigable web interface — no backend, no CMS, no additional infrastructure.

This document covers the initial MVP scope. Future phases covering search, authentication, and Claude-powered document summarisation are tracked separately.

2 Objectives

  • Provide a single URL where all Tymraft internal documents can be found and accessed.
  • Eliminate manual document management — any file committed to the repo should appear automatically.
  • Support folder-based organisation so teams can structure their own documentation areas.
  • Render documents in a consistent, readable format without requiring authors to write HTML.
  • Integrate with the existing Claude project setup to allow AI-assisted document creation and updates.

3 Scope

In scope

  • A browsable file explorer at the root of tymraft.github.io rooted at the docs/ directory.
  • Folder navigation with breadcrumb trail and live fetching from the GitHub API.
  • Support for Markdown (.md) and HTML (.html) document types in the initial release.
  • A consistent document chrome (top bar, back navigation, metadata) for HTML documents.
  • Claude plugin infrastructure: GETTING_STARTED.md, plugin.json, and skill files hosted in the same repo.

Out of scope

  • Full-text search across all documents (Phase 2).
  • Authentication or access control — the repository is public.
  • A WYSIWYG document editor in the browser.
  • Versioning UI — Git history serves this purpose natively.
  • PDF rendering or preview.

4 Requirements

Functional requirements

ID Requirement Priority
FR-01 The explorer shall display all files and folders within docs/ by fetching the GitHub Contents API at page load. High
FR-02 Clicking a folder shall navigate into that folder and update the breadcrumb trail accordingly. High
FR-03 Clicking a file shall open it at its canonical tymraft.github.io URL. High
FR-04 The explorer shall support filtering by file name within the current folder via a search field. Medium
FR-05 Columns shall be sortable by name, last modified date, and file size. Medium
FR-06 HTML documents shall include a consistent top bar with a back link, breadcrumb path, and document status badge. Medium
FR-07 The GETTING_STARTED.md file shall contain instructions that Claude can follow to self-install the Tymraft plugin and skills. High

Non-functional requirements

  • Performance: Folder contents shall load within 2 seconds on a standard connection. No server-side rendering; all fetching is client-side.
  • Compatibility: The explorer shall function correctly on the latest versions of Chrome, Safari, Firefox, and Edge.
  • Maintainability: All documents and configuration are managed via Git commits; no database or admin panel is required.
  • Accessibility: The explorer shall use semantic HTML and maintain visible keyboard focus states.

7 Risks

GitHub API rate limits: Unauthenticated requests are capped at 60/hour per IP. For a small internal team this is unlikely to be an issue, but heavy use may require adding an auth token to API calls.
RiskLikelihoodImpactMitigation
GitHub API rate limiting Low Medium Add optional GitHub PAT to API requests if rate limits become an issue.
Repo visibility change Low High Document that making the repo private will break the public API calls in the explorer.
File rendering inconsistency Medium Low Establish a standard HTML template (this document) for all rich documents.
